# 分布式事务优化方案说明

## 一、方案概述

### 1.1 问题分析

在高并发场景下，使用 Seata 的 `@GlobalTransactional`（AT 模式）存在以下问题：

- **性能瓶颈**：全局锁机制导致性能下降
- **单点故障**：事务协调器成为瓶颈
- **网络开销**：频繁的事务协调通信
- **不适合高并发**：无法支撑高QPS场景

### 1.2 解决方案

采用**最终一致性 + 补偿机制**方案：

- **事件驱动架构**：通过消息队列实现服务解耦
- **本地事务**：每个服务使用本地事务保证数据一致性
- **补偿机制**：通过补偿操作保证最终一致性
- **状态机管理**：管理订单状态流转
- **幂等性保证**：防止重复处理

## 二、架构设计

### 2.1 订单创建流程

```
用户请求
  ↓
订单服务（创建订单，状态：待支付）
  ├─ 本地事务：保存订单和订单明细
  └─ 发布事件：OrderCreatedEvent
  ↓
┌─────────────────┬─────────────────┬─────────────────┐
│  库存服务       │  优惠券服务     │  支付服务       │
│  (异步处理)     │  (异步处理)     │  (异步处理)     │
│  扣减库存       │  锁定优惠券     │  预支付         │
│  发送确认消息   │  发送确认消息   │  发送确认消息   │
└─────────────────┴─────────────────┴─────────────────┘
  ↓
订单服务（接收确认，更新订单状态）
```

### 2.2 补偿机制

如果某个服务处理失败：
1. 发送补偿消息
2. 执行补偿操作（回滚库存、释放优惠券等）
3. 更新订单状态为失败

## 三、核心组件

### 3.1 事件框架

- `BaseEvent`: 事件基类
- `EventPublisher`: 事件发布器
- `AbstractEventListener`: 事件监听器基类
- `EventListener`: 事件监听器接口

### 3.2 订单状态机

- `OrderStatus`: 订单状态枚举
- `OrderStateMachine`: 订单状态机接口
- `OrderStateMachineImpl`: 订单状态机实现
- `OrderStateLog`: 订单状态流转记录

### 3.3 补偿服务

- `OrderCompensationService`: 订单补偿服务
- `InventoryCompensationService`: 库存补偿服务
- `CouponCompensationService`: 优惠券补偿服务（待实现）

### 3.4 幂等性保证

- `IdempotentUtils`: 幂等性工具类
  - 使用 Redis 实现分布式锁
  - 使用业务ID防止重复处理

## 四、事件定义

### 4.1 订单相关事件

- `OrderCreatedEvent`: 订单创建事件
- `OrderCancelledEvent`: 订单取消事件

### 4.2 库存相关事件

- `InventoryDeductedEvent`: 库存扣减成功事件
- `InventoryDeductFailedEvent`: 库存扣减失败事件

## 五、使用示例

### 5.1 订单创建

```java
@Transactional(rollbackFor = Exception.class)
public OrderDTO createOrder(OrderCreateDTO createDTO) {
    // 1. 创建订单（本地事务）
    Order order = createOrderEntity(createDTO);
    orderMapper.insert(order);
    
    // 2. 发布订单创建事件
    OrderCreatedEvent event = OrderCreatedEvent.builder()
            .orderId(order.getId())
            .orderNo(order.getOrderNo())
            .build();
    eventPublisher.publishOrderEvent("order-create", event);
    
    return convertToDTO(order);
}
```

### 5.2 库存扣减（事件监听）

```java
@Component
public class OrderCreatedListener extends AbstractEventListener<OrderCreatedEvent> {
    
    @Override
    protected void doHandle(OrderCreatedEvent event) {
        // 扣减库存
        inventoryService.deductStock(...);
        
        // 发送成功事件
        eventPublisher.publishInventoryEvent("inventory-deducted", successEvent);
    }
}
```

### 5.3 补偿处理

```java
@Component
public class InventoryDeductFailedListener extends AbstractEventListener<InventoryDeductFailedEvent> {
    
    @Override
    protected void doHandle(InventoryDeductFailedEvent event) {
        // 触发订单补偿
        orderCompensationService.compensateOrderCreation(
                event.getOrderId(), 
                event.getOrderNo(), 
                event.getReason()
        );
    }
}
```

## 六、优势

1. **高性能**：异步处理，无全局锁
2. **高可用**：服务解耦，单点故障不影响整体
3. **可扩展**：易于添加新的服务参与流程
4. **最终一致性**：通过补偿机制保证数据最终一致
5. **可观测性**：消息和状态可追踪

## 七、注意事项

1. **幂等性**：所有事件处理必须保证幂等性
2. **补偿机制**：必须实现完整的补偿逻辑
3. **状态管理**：使用状态机管理订单状态流转
4. **监控告警**：监控消息积压和补偿执行情况
5. **异常处理**：妥善处理各种异常情况
